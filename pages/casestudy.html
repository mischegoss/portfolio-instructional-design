My Role

I owned this project end-to-end: architecture, infrastructure, testing, and iteration.

Why I Built It This Way

The core decisions were driven by a single requirement: every answer had to come from the documentation and support knowledge base only. No improvisation, no outside sources.

The Problem

Enterprise documentation at scale creates a self-service gap. Users have questions. The answers exist somewhere in the docs. But somewhere isn't good enough when you're trying to resolve a critical issue or find a solution fast.

As a result, users were turning to the support team to find answers that existed in the docs. This increased the burden on support staff and pushed resolution times out, sometimes by days. The out-of-the-box solution made it worse: a general-purpose chatbot purchased from a vendor struggled with deeply nested documentation and highly technical language, producing misleading or outright hallucinated answers at a 27% fail rate even on simple requests. The vendor also had no way to surface the richer step-by-step content in the support knowledge base, leaving a significant portion of available answers unreachable except by contacting support.

The Constraint That Mattered

It was clear that a general-purpose chatbot wasn't the answer. The consequence of a hallucination — a confident, plausible, wrong answer — can be dire when documentation is aimed at IT professionals working on enterprise systems. A user following bad instructions can break a production workflow and set them back for days.

Design Requirement

Every answer had to be grounded strictly in versioned product documentation. No improvisation. No answers from outside the knowledge base. The bot needed to be grounded in the actual docs.

The Approach

This was the second generation of the chatbot. The original vendor solution never performed well enough to be reliable, so I spent time iterating on and improving it across four product lines before ultimately replacing it entirely. This failed experiment gave me crucial insights into the limitations of the current model and the features users actually needed. Then I rebuilt the chatbots from the ground up in Python on Google Cloud.

The custom solution uses a RAG (retrieval-augmented generation) architecture, retrieving answers from the actual docs before generating a response, then surfacing inline source citations so users can verify exactly where the answer came from and navigate directly to the relevant page.

For deeper troubleshooting, the chatbot integrates with the Zendesk API to search support articles, version-aware so a user on version 8.0 gets results tied to their version and not a newer release that might not apply.

Making It Reliable

Shipping it was one problem. Trusting it was another. I ran adversarial testing to surface failure patterns including edge cases, prompt gaps, and retrieval failures before users could find them first. I also implemented a thumbs up/down feedback loop so real usage data could drive continuous improvements after launch.
