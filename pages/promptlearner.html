The principle

The model follows what is written in the intent description, not what is implied.

TLDR

Following the recent LLM migration, intent prediction began silently failing because many of our existing intent descriptions had been written assuming the previous ChatGPT model would fill in missing context. The new model interprets description language more literally. As a result, when descriptions were ambiguous, intent matching failed more often, and those failures are surfacing as support tickets that have been sent to content. This has resulted in a lengthy resolution time per ticket.

This guide gives you a structured framework for writing intent descriptions that stay reliable across model changes, and for fixing the tickets where the intent description is silently failing.

Diagnostics

Signs a description may be leaning on inference

In the description

  • No explicit action and object — the model has to parse them
  • Example phrases are all complete sentences, no minimal or fragmented inputs
  • "Do not use" is missing, short, or implied by wording
  • Uses words like "relevant," "appropriate," or "general" without definition

In the ticket queue

  • Users say the KB used to return the right answer
  • Same description is getting patched for different failures
  • Misroutes cluster around specific phrasing patterns
  • Fixing one user's phrasing breaks matching for another
The prompt that solves it

A structured prompt for durable descriptions

Here is the prompt you can use to generate or rewrite an intent description. Paste it into your AI model interface, replace [system name], fill in the intent name and plain description at the bottom, and run. If the model asks you for a missing input, provide it before continuing.

/* ROLE ASSIGNMENT */ You are generating an intent description for [system name], an AI-powered knowledge base that matches user queries to content using intent prediction. The system requires clear, explicit, and complete inputs to route correctly. /* STAGED LOGIC */ STAGE 1: VALIDATE INPUT Check all conditions before generating output: 1. Intent name is present. 2. A clear ACTION and OBJECT are defined. 3. Description is specific enough to avoid matching unrelated intents. If any condition is missing, ask only the missing questions. Do not generate output. STAGE 2: GENERATE DESCRIPTION You may use words from the input, direct synonyms, and common automation terms: webhook, workflow, automation, API, request, trigger, action, process. Only include logic explicitly stated or clearly implied by the action-object pair. /* EXPLICIT RULES */ RULE 1: DEFINITION Write 1-2 sentences defining what the intent does and when it is used. RULE 2: TRIGGER COVERAGE Write a short "Use this intent when..." section. Use verbs: trigger, run, start, execute, launch, invoke, call, fire. Apply to action + object in combined phrases. Do not list separately. RULE 3: EXAMPLES (generate 6-10 phrases total, do not exceed 10) Generate user phrases across four registers: - Formal sentences (2-3) - Short commands (2-3) - Minimal inputs (1-2) - Same intent, different word order (1-2) Keep phrases short. RULE 4: WHEN TO USE One paragraph stating when this intent should fire. RULE 5: DO NOT USE State when this intent must NOT fire. Always include: Do not use for informational queries, status checks, or non-action requests. RULE 6: STYLE Simple language. Short sentences. No marketing tone. /* OUTPUT FORMAT CONTROL */ RULE 7: OUTPUT FORMAT Return only plain text. No markdown. No explanations. Output must include exactly these sections in this order: Definition / Trigger Coverage / Examples / When To Use / Do Not Use /* INPUTS */ Intent name: [INSERT INTENT NAME] Plain description: [INSERT PLAIN DESCRIPTION]
How the prompt fixes what broke

What each part does, and why it matters

The prompt has five structural jobs. Each one replaces something the old descriptions had left for the model to infer — and each one targets a specific failure mode that surfaced after the migration.

  1. 01
    Role assignment — the top block
    Tells the LLM this is an intent description for a routing system, not a blurb or a summary. Fixes: outputs that drift in tone or read like marketing copy.
  2. 02
    Staged logic — Stage 1 validation
    Checks that an intent name, a clear action and object, and sufficient specificity are all present before generating anything. If something is missing, the model asks. Fixes: descriptions generated from incomplete inputs, where the old model silently filled the gap and the new model will not.
  3. 03
    Explicit rules — Rules 1 through 6
    Each section of the description gets its own rule: definition, trigger coverage, four registers of examples, when-to-use, do-not-use, and style. Fixes: thin descriptions that only worked because the old model inferred the parts that were not written.
  4. 04
    Output format control — Rule 7
    Plain text, fixed section order, no markdown, no preamble. Fixes: formats that drifted run to run and made descriptions hard to compare or parse downstream.
  5. 05
    Constraint framing — woven through the whole prompt
    Not a single block. Every rule quietly optimizes for the same things: specificity, coverage across phrasings, and non-overlap with neighboring intents. Fixes: descriptions that looked fine individually but collided at match time.
Applying this pattern elsewhere

The same structure works for other prompt tasks

The five structural jobs are not specific to intent descriptions. They are a general recipe for getting consistent output from an LLM any time ambiguity would cost you. Next time you are writing a prompt for a different task, walk through the same five questions:

  1. 01
    What is the role?
    What kind of output are you asking for, and for what system? State it explicitly at the top.
  2. 02
    What inputs are required?
    What can't the model do its job without? Add a validation stage so missing inputs trigger a question, not a guess.
  3. 03
    What must the output contain?
    Every section, every field, every behavior you care about. Spell each one out as a rule — each rule removes a decision the model would have made alone.
  4. 04
    What is the output format?
    Who or what consumes this downstream? Match that system's requirements exactly — plain text, JSON, section order, all of it.
  5. 05
    What should the output optimize for?
    Completeness, specificity, brevity, a single correct answer? Name it up front so the model has a goal to frame every other rule against.
Places this pattern transfers directly: ticket categorization, content tagging, knowledge extraction, routing logic, structured summaries — anywhere reliable, comparable output matters more than creative variation.
