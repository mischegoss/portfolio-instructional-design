Prompt engineering:
when the model changes

Format 30-min training + handout
Audience Support Engineers
Context Model migration (e.g., GPT-4o → GPT-4.1–class systems)
Role Instructional Designer · Documentation Engineer
The Design Challenge

A model change caused a silent failure in intent prediction. Output degraded without errors. The issue was not the intent descriptions themselves: it was how they were written. They relied on the model to infer missing context. This training converts that failure into a repeatable prompt engineering framework.

The Problem
// incident summary
systemAI-powered knowledge base
componentIntent prediction
triggerModel migration
failure_modeMissing or incomplete routing output (no error surfaced)

root_causePrompts relied on implicit inference instead of explicit inputs

Models vary in how they handle incomplete inputs: some tolerate gaps, others do not. The failure was not in the descriptions—it was in assuming the model would fill in gaps.

Root cause in one line

The model follows what is written, not what is implied.

The Solution: Structured Prompt Engineering

Structured prompting replaces implicit assumptions with explicit requirements. The prompt becomes the system specification. A reliable prompt has five components:

# Component Function
01 Role assignment Sets context for how the model interprets the task. A model given routing context behaves differently than one without it.
02 Constraint framing Defines what the output must optimize for. Shifts the model's priority from readability to completeness and coverage.
03 Staged logic Guard clause. Stage 1 checks that required inputs are present. If correctly implemented, the model should ask for clarification rather than generating on incomplete inputs.
04 Explicit rules Clear, non-conflicting rules reduce variation and improve consistency at scale. Each rule removes a model decision.
05 Output format control If output feeds a downstream system, format is a requirement, not a preference. Format violations are data quality issues.
Core principles
Do not rely on model inference: encode requirements explicitly
Validate inputs before generating output
Constrain structure and format to system requirements
Design for consistency over readability
Model behavior may change; system requirements should not
The Full Prompt

Designed for models that require explicit instructions and do not reliably infer missing context. Performance may vary across models. Replace [system name] and complete the input fields before use.

/* ROLE ASSIGNMENT */ You are generating an intent description for [system name], an AI-powered knowledge base that matches user queries to content using intent prediction. The system requires clear, explicit, and complete inputs to route correctly. /* STAGED LOGIC */ STAGE 1: VALIDATE INPUT Check all conditions before generating output: 1. Intent name is present. 2. A clear ACTION and OBJECT are defined. 3. Description is specific enough to avoid matching unrelated intents. If any condition is missing, ask only the missing questions. Do not generate output. STAGE 2: GENERATE DESCRIPTION You may use words from the input, direct synonyms, and common automation terms: webhook, workflow, automation, API, request, trigger, action, process. Only include logic explicitly stated or clearly implied by the action-object pair. /* EXPLICIT RULES */ RULE 1: DEFINITION Write 1-2 sentences defining what the intent does and when it is used. RULE 2: TRIGGER COVERAGE Write a short "Use this intent when..." section. Use verbs: trigger, run, start, execute, launch, invoke, call, fire. Apply to action + object in combined phrases. Do not list separately. RULE 3: EXAMPLES (generate 6-10 phrases total, do not exceed 10) Generate user phrases across four registers: - Formal sentences (2-3) - Short commands (2-3) - Minimal inputs (1-2) - Same intent, different word order (1-2) Keep phrases short. RULE 4: WHEN TO USE One paragraph stating when this intent should fire. RULE 5: DO NOT USE State when this intent must NOT fire. Always include: Do not use for informational queries, status checks, or non-action requests. RULE 6: STYLE Simple language. Short sentences. No marketing tone. /* OUTPUT FORMAT CONTROL */ RULE 7: OUTPUT FORMAT Return only plain text. No markdown. No explanations. Output must include exactly these sections in this order: Definition / Trigger Coverage / Examples / When To Use / Do Not Use /* INPUTS */ Intent name: [INSERT INTENT NAME] Plain description: [INSERT PLAIN DESCRIPTION]
Common failure patterns
Missing action or object → validation triggers; resolve before continuing
Too few examples → poor query matching coverage
Overly broad description → false positives
Validation step omitted → incomplete inputs generate invalid output
Incorrect format → output cannot be used downstream

How to run it

1Open your AI model interface.
2Replace [system name] and complete the input fields at the bottom.
3Submit the prompt.
4If a clarification is requested, provide the missing input before proceeding.
5Verify all five sections are present and correctly ordered.
Takeaway

Prompts fail when they rely on inference. Explicit structure—role, constraints, validation, rules, and format—encodes system requirements and improves reliability across model changes.
